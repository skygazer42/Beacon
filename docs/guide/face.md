# 人脸库管理

当前页面是 Analyzer 人脸向量库的管理界面，不是独立的人员档案或多层“人脸库”系统。Admin 会把操作代理到 Analyzer（默认 `127.0.0.1:9993`）。

## 前置条件

1. Analyzer 已启动，Admin 配置的 `analyzerHost` 和 `analyzerPort` 可达。
2. 需要用图片生成特征时，已配置可用的特征算法编码；否则直接提供 Embedding JSON。
3. 生产环境已经为生物特征数据确定访问、留存、删除和备份制度。

页面顶部会显示直连接口、回退数据和搜索状态。出现“直连接口不可用”时，先修复 Analyzer 链路；回退数据只用于展示，不代表新增、搜索或删除动作可用。

## 已实现的操作

| 操作 | 当前行为 |
|---|---|
| 列表 | 读取人脸条目、向量维度和搜索开关状态 |
| 本地筛选 | 按 ID、名称等文本搜索，并按返回数据中的分组字段筛选 |
| 新增 | 提交唯一 ID、可选名称，以及 Embedding JSON 或 Image Base64 |
| 相似搜索 | 使用 Embedding JSON 或 Image Base64，可选特征算法和最小分数 |
| 删除 | 按条目 ID 删除 |
| 启用 / 停用搜索 | 控制 Analyzer 人脸搜索功能 |

当前页面没有人员详情页、多照片档案、CSV/文件夹批量导入、1:1 双图比对或比对历史标签页；不要按这些未实现能力设计验收流程。

## 新增与搜索

- **Embedding JSON**：数值数组的维度必须与当前人脸库一致。
- **Image Base64**：后端调用指定或默认特征算法提取向量；只填写裸 Base64 内容，不要把敏感图片复制到 Issue。
- **特征算法编码**：例如部署中实际存在的人脸特征模型编码；示例名称不代表仓库附带该模型。
- **最小分数**：相似搜索的命中阈值，实际范围和方向以 Analyzer 返回契约为准。

## 排查

1. `curl http://127.0.0.1:9993/api/health` 验证 Analyzer，而不是只检查 9991 页面。
2. 检查 Admin 到 Analyzer 的地址、鉴权和防火墙。
3. 图片方式失败时，改用已知维度的 Embedding 验证数据库本身，再检查特征算法。
4. 维度不一致时不要混用不同特征模型产生的向量。
5. 删除和重建库前先确认数据备份及合规要求。
